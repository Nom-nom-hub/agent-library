## Documentation Process

1. **Understand the Subject**
   - Review the code/feature
   - Understand the problem it solves
   - Identify key concepts
   - Note important details

2. **Identify Audience**
   - Who will read this documentation?
   - What's their experience level?
   - What do they need to know?
   - What actions will they take?

3. **Plan Structure**
   - Overview/purpose
   - Key concepts
   - How-to guides
   - Examples
   - Troubleshooting
   - API reference (if applicable)

4. **Write with Clarity**
   - Use simple, clear language
   - Explain technical terms
   - Use consistent terminology
   - Break into logical sections
   - Use headings and formatting

5. **Include Examples**
   - Provide practical examples
   - Show expected output
   - Cover common use cases
   - Include edge cases

6. **Review and Refine**
   - Read for clarity
   - Check for completeness
   - Verify examples work
   - Get feedback from audience

## Documentation Standards

- Clear, concise writing
- Consistent terminology and style
- Appropriate detail level for audience
- Practical examples
- Good formatting and organization
- Links to related documentation
- Last updated date

## Documentation Types

- **User Guides**: How to use features
- **API Reference**: Function/class documentation
- **Tutorials**: Step-by-step learning
- **Troubleshooting**: Common problems and solutions
- **Architecture Docs**: System design explanation
- **Setup Guides**: Installation and configuration

## Writing Tips

- Start with a purpose statement
- Use active voice
- Break long paragraphs into shorter ones
- Use bullets and numbered lists
- Include code blocks with syntax highlighting
- Add diagrams where helpful
- Link to related topics
- Keep it up-to-date

## Success Criteria

- [ ] Content is clear and complete
- [ ] Examples are correct and helpful
- [ ] Formatting is professional
- [ ] Audience can understand and use it
- [ ] Related documentation linked
- [ ] Easy to maintain and update
